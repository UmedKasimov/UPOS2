# ТЗ: доработка CRM и её связей с клиентами, телефонией и мессенджерами

Версия: 1.0 · Дата: 2026-06-27 · Статус: на согласование

## 1. Цель

Превратить текущий CRM-канбан в связную воронку, где клиент, сделка/задача,
звонок и переписка — это связанные сущности с единым таймлайном активностей,
автоматическим созданием лидов из входящих контактов и контролем сроков (SLA).

## 2. Область

Затрагиваемые модули: `crm`, `clients` (counterparties), `telephony`, `messengers`.
Файлы-ориентиры: `db_models.py`, `main.py` (роуты `/crm/*`, `/telephony/*`, `/messengers/*`,
`/api/*`), `templates/_business_module_crm.html`, `static/crm-kanban.js`,
`pg_schema_align.py`, `schema_postgres.sql`.

## 3. Изменения модели данных

### 3.1 `crm_records` (расширить)
- `due_date`: перевести строковое поле на реальную дату (`Date`/`DateTime`),
  сохранить обратную совместимость при чтении старых строковых значений.
- Добавить поля: `responsible_user_id` (FK→users, nullable), `lead_source`,
  `stage_id` (вынести из JSON в колонку для фильтра/индексов), `last_activity_at`,
  `next_action_at` (для SLA/напоминаний).
- Индексы: `(workspace_owner_id, stage_id)`, `(workspace_owner_id, counterparty_id)`,
  `(workspace_owner_id, next_action_at)`.

### 3.2 Новая таблица `crm_activities` (единый таймлайн)
Поля: `id`, `workspace_owner_id`, `counterparty_id` (nullable),
`crm_record_id` (nullable, FK→crm_records SET NULL), `kind`
(`note|call|message|stage_change|status_change|task|system`),
`channel` (`telegram|whatsapp|instagram|sip|manual|...`), `direction`
(`in|out|system`), `title`, `body`, `payload` (JSONB: ссылки на внешние id,
вложения, длительность звонка и т.п.), `actor_user_id` (nullable),
`occurred_at`, `created_at`.
Индексы: `(workspace_owner_id, counterparty_id, occurred_at)`,
`(workspace_owner_id, crm_record_id, occurred_at)`.

> Телефония и мессенджеры пишут активности сюда, а карточка/клиент читают
> единую ленту. Это снимает зависимость от ephemeral-данных в `workspace_settings`.

### 3.3 `counterparties` (контакты)
- В `data` (JSONB) формализовать контакты: `messengers: [{channel, handle, chat_id}]`,
  нормализованный `phone_e164`. Дедупликация контакта по `phone_e164` и по паре
  `(channel, chat_id)`.

Все изменения — через `Base.metadata.create_all` + `pg_schema_align`
(добавление колонок idempotent), плюс правка `schema_postgres.sql` как DDL-ориентир.

## 4. Бэкенд / API

### 4.1 CRM
- `POST /crm/save` — добавить режим редактирования (`record_id` опционально),
  принимать `counterparty_id` явно (не только имя), писать `stage_change` в активности.
- `GET /crm/{record_id}` — JSON с карточкой + таймлайн активностей (для детального модала).
- `POST /crm/{record_id}/activity` — добавить заметку/активность вручную.
- `/crm/{record_id}/stage` — уже есть; дополнительно писать `crm_activities(kind=stage_change)`.
- Список карточек: пагинация/лимит, серверная фильтрация по `stage_id`,
  `responsible_user_id`, дате.

### 4.2 Телефония → CRM
- `/api/telephony/incoming-call`:
  - нормализация телефона в E.164, поиск/создание `Counterparty`,
  - запись `crm_activities(kind=call, channel=sip, direction=in)`,
  - опц. авто-создание лида (если нет открытой сделки у клиента) — по настройке.
  - защита вебхука токеном (`X-UPOS-Webhook-Token` из настроек воркспейса).
- Карточка CRM/клиента: кнопка «Позвонить» → `POST /api/telephony/click-to-call`
  (если провайдер поддерживает) либо `tel:`-ссылка как fallback.

### 4.3 Мессенджеры → CRM
- Заменить URL-префилл на `POST /crm/from-chat` (передаёт `channel`, `chat_id`,
  `counterparty_id`): резолвит/создаёт контакт, дедуп по `(channel, chat_id)`,
  создаёт сделку и привязывает переписку (`crm_activities(kind=message)`).
- Входящее сообщение (Telegram webhook, уже есть инфраструктура `telegram_*`):
  писать `crm_activities(kind=message, direction=in)`; по настройке — авто-лид.
- Привязка чата к существующей сделке из карточки.

### 4.4 Клиенты ↔ CRM
- В карточке клиента — панель «Сделки и задачи» + единый таймлайн (`crm_activities`).
- Жёсткая связь по `counterparty_id`; имя оставить как отображение/поиск.

## 5. UI / UX
- Детальный модал карточки сделки: поля + таймлайн (звонки/сообщения/заметки),
  кнопки «Позвонить», «Открыть чат», «Добавить заметку», «Редактировать».
- На канбан-карточке: индикатор просрочки (`next_action_at`), канал последнего контакта.
- В карточке клиента: вкладка «CRM» со сделками и лентой активностей.
- Фильтры канбана выполняются на сервере (сейчас часть — на клиенте).

## 6. Автоматизация (по настройкам воркспейса)
- Авто-создание лида на первый входящий звонок/сообщение от неизвестного контакта.
- Авто-назначение ответственного (правило: автор/дежурный/по источнику).
- SLA: подсветка и Telegram-напоминание при наступлении `next_action_at`
  (через существующий `telegram_scheduler`/`telegram_notifier`).
- Уведомление ответственному при назначении/смене этапа.

## 7. (Опц.) Связь сделки с деньгами
- Привязка сделки к продаже/счёту (`sale_documents`/`payment_documents`),
  агрегаты по этапам брать из связанных документов, а не из свободного `amount`.

## 8. Фазы внедрения
1. **Фундамент:** `crm_activities`, `due_date`→дата, `counterparty_id` явно,
   редактирование записи, детальный модал с таймлайном.
2. **Телефония:** вебхук пишет активности + авто-лид + защита токеном; кнопка звонка.
3. **Мессенджеры:** `POST /crm/from-chat`, входящие сообщения в таймлайн, авто-лид.
4. **Клиент-360:** панель CRM и таймлайн в карточке клиента.
5. **Автоматизация/SLA/уведомления.**
6. **(Опц.) Связь с деньгами.**

## 9. Нефункциональные требования
- Изоляция по `workspace_owner_id` во всех новых запросах.
- CSRF на всех изменяющих формах/JSON; токен на вебхуках.
- Idempotent миграции схемы; без даунтайма.
- Тесты — ручные сценарии (в проекте нет автотестов), плюс smoke по новым роутам.

## 9a. Статус реализации

**Фаза 1 — реализована (2026-06-27):**
- Новая таблица `crm_activities` (модель в `db_models.py`, авто-создание через `create_all`).
- `crm_records`: добавлены колонки `stage_id`, `responsible_user_id`, `next_action_at`,
  `last_activity_at` (миграция через `sync_missing_public_columns` + патчи в
  `pg_schema_align._REQUIRED_COLUMN_PATCHES`).
- Бэкенд: `crm_save` — режим редактирования (`record_id`) + явный `counterparty_id`
  + лог активностей; `crm_stage_update` — лог смены этапа + запись `stage_id`;
  новые роуты `GET /crm/{record_id}` (карточка + таймлайн) и
  `POST /crm/{record_id}/activity` (заметка/звонок/сообщение).
- UI: детальный модал с таймлайном и формой заметки, кнопка «Редактировать»
  (префилл формы), бейдж «Просрочено» на карточке, кликабельные карточки/строки.

**Решение по `due_date`:** колонку оставили строковой (обратная совместимость с
шаблонами), а для SLA добавили отдельную дату-колонку `next_action_at` (заполняется
из `due_date` при сохранении; для старых записей работает fallback-парсинг строки).

**Фаза 2 — телефония → CRM (реализована):**
- Общие хелперы `_crm_find_open_deal_for_counterparty`, `_crm_ensure_lead_for_counterparty`,
  `_crm_register_contact_event` (привязка события к открытой сделке + авто-лид).
- `/api/telephony/incoming-call`: защита токеном (`telephony_webhook_token` в настройках,
  заголовок `X-UPOS-Webhook-Token`), поиск/создание клиента по телефону, запись
  активности `call/sip/in`, авто-лид (настройка `telephony_auto_lead`, по умолчанию вкл.).
- `telephony_call_save` (ручной звонок): привязка к клиенту + активность + авто-лид.
- `POST /api/telephony/click-to-call`: логирует исходящий звонок; кнопка «Позвонить»
  в детали CRM вызывает его и открывает `tel:`.

**Фаза 3 — мессенджеры → CRM (реализована):**
- `POST /crm/from-chat` заменил URL-префилл: резолв/создание контрагента по
  (канал, chat_id/handle/телефон) с дедупом, авто-лид, активность `message`.
  `_crm_messenger_counterparty` хранит привязки в `counterparty.data.messengers`.
- `messengers.js`: кнопка «в CRM» шлёт POST и открывает созданную сделку.

**Фаза 4 — клиент-360 (реализована):**
- В карточку клиента (`_client_card_context`) добавлены события из `crm_activities`
  (звонки/сообщения/заметки) в общий таймлайн «История». CRM-статус клиента
  синхронизируется с этапом авто-лида.
- Известный минор: лог звонков дублируется (новые `crm_activities` + старые записи
  `telephony_calls` в JSON) — косметика, не критично.

**UX/дизайн (реализовано):**
- Плавающий телефон (call FAB) — глобальная кнопка для авторизованных
  (`base.html`, `static/telephony-fab.js`): панель-дайлпад, ввод номера, «Позвонить»
  (→ `POST /api/telephony/click-to-call`, матчинг клиента по телефону) + `tel:`.
- «2026-рефреш» поверх токенов (конец `static/styles.css`): градиентные кнопки,
  мягкие тени/подъём карточек, focus-ring, тонкие скроллбары, глассморфизм панели.

**Найдено в логах:** внешний источник шлёт `POST /api/telephony/calls/ingest`
(→ 302 /auth, маршрута нет). Похоже на реальную АТС/телефонию — кандидат на
алиас к `/api/telephony/incoming-call` (нужен формат payload провайдера).

**Дальше (не реализовано):** автоматизация/SLA-напоминания (фаза 5, через
`telegram_scheduler`/`telegram_notifier`), связь сделки с деньгами (фаза 6).
